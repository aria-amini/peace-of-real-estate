Dashboard Page Layout API

Design principle. Put all generic dashboard primitives in a single components/dashboard.tsx module, split into comment-sections. Keep agent/consumer-specific config (nav items, labels, guards) co-located in route folders. No variant props — routes pass their specific values into the shared API.

Original problem

  • consumer/dashboard/matches.tsx and agent/dashboard/introductions.tsx wrap themselves in *SidebarShell even though the parent dashboard.tsx layout already provides the sidebar, nesting SidebarProvider.
  • Every dashboard page copy-pastes the viewport-centering formula xl:ml-[calc((100vw-...)/2-var(--sidebar-width))].
  • Adding a page requires recreating padding, mobile trigger, and centering math.

Desired behavior

Below xl, content centers inside the page area next to the sidebar. At xl+, the same content box centers within the full viewport, offsetting the sidebar.

Proposed API

Single generic module

Create src/components/dashboard.tsx with sections:

// ============================================================
// Types
// ============================================================
export type SidebarItem = { ... }

// ============================================================
// Layout shell
// ============================================================
export function DashboardShell({ sidebar, children }) { ... }

export function DashboardSidebar({ items, profileLabel, profileHint }) { ... }

// ============================================================
// Page shell
// ============================================================
export function DashboardPage({ children, className }) { ... }

export function DashboardPageMobileNav({ label }) { ... }

export function DashboardPageHeader({ icon, eyebrow, title, description, actions }) { ... }

// ============================================================
// Shared dashboard widgets
// ============================================================
export function StatusRow({ ... }) { ... }

Route-specific sidebars

Configured sidebars live next to the routes that need them:

// routes/(app)/agent/dashboard/-components/sidebar.tsx
export function AgentSidebar() {
  return (
    <DashboardSidebar
      items={agentItems}
      profileLabel="Agent dashboard"
    />
  )
}

Layout routes

Each role layout just composes the generic shell with its own sidebar: 

// routes/(app)/agent/dashboard.tsx
<DashboardShell sidebar={<AgentSidebar />}>
  <Outlet />
</DashboardShell>

Page usage

Pages get blank space and handle their own content width:

<DashboardPage>
  <DashboardPageMobileNav label="Account menu" />
  <div className="mx-auto w-full max-w-5xl">
    {/* content */}
  </div>
</DashboardPage>

Centering math

DashboardPage applies:

w-full px-6 py-10 xl:-ml-[calc(var(--sidebar-width)/2)]

The SidebarInset center is offset right by half the sidebar. Shifting the page left by the same amount makes any mx-auto max-w-* child viewport-centered at xl+.

Commit 1: Centralize auth redirects in __root

Goal: one place decides who can enter dashboard routes.

Use a hybrid approach: broad redirects in __root, with per-route overrides possible but rarely used.

After the beta access check, add session-based redirects for protected prefixes: 

const protectedPrefixes = ['/agent/dashboard/', '/consumer/dashboard/']

if (!session && protectedPrefixes.some(p => location.pathname.startsWith(p))) {
  if (location.pathname.startsWith('/agent/')) {
    throw redirect({ to: '/login', search: { redirect: location.pathname } })
  }
  if (location.pathname.startsWith('/consumer/')) {
    throw redirect({ to: '/consumer/signup', search: { step: 'intro' } })
  }
}
  • Add the session check and protected-prefix redirects to src/routes/__root.tsx.
  • Remove beforeLoad from src/routes/(app)/agent/dashboard.tsx.
  • Remove beforeLoad session check from src/routes/(app)/consumer/dashboard/matches.tsx.
Verify: unauthenticated users are redirected for /agent/dashboard/* and /consumer/dashboard/*; authenticated users still reach pages.
